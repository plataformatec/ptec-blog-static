Ecto 3.0 is just around the corner and as you may already know it reached stable API<\/a>. To make sure everything works properly I thought I’ll try updating one of the first projects that ever used Ecto: Hex.pm<\/a>.<\/p>\n

The whole upgrade was done in a single pull request<\/a>, which we will break down below.<\/p>\n

First, the required steps:<\/p>\n

    \n
  1. Update mix.exs to depend on ecto_sql<\/code> and bump the postgrex dependency. Note: SQL handling have been extracted out into a separate ecto_sql<\/code><\/a> project, so we need to add that new dependency. (6b3b78cf<\/code><\/a>)<\/li>\n
  2. DBConnection 2.0 no longer ships with Sojourn and Poolboy pools, so we can remove the pool<\/code> configuration and use the default pool implementation. (760026f3<\/code><\/a>)<\/li>\n
  3. Speaking of pools, we need to make sure pool_size<\/code>\u00a0is at least 2<\/code>\u00a0when running migrations.<\/li>\n
  4. JSON library is now set on the adapter and not on Ecto (e16ebd8f<\/code><\/a>) and because we were already using the recommended package, Jason<\/a>, we don’t need that configuration anymore. (66f9cbdf<\/code><\/a>)<\/li>\n
  5. Ecto 3.0 makes date\/time handling stricter with regards to precision. So we need to either update the types of our schema fields or make sure we truncate date\/time values. For example, when we define a field as time<\/code> we can’t put value with microsecond precision and similarly we can’t put into a time_usec<\/code> field a value without microsecond precision. (2e34b833<\/code><\/a>)<\/li>\n
  6. Constraint handling functions like Ecto.Changeset.unique_constraint\/3<\/code><\/a> are now including in the error metadata the type and the name of the constraint, which broke our test that was overly specific. (3d19f903<\/code><\/a>)<\/li>\n<\/ol>\n

    Secondly, we got a couple deprecation warnings so here are the fixes:<\/p>\n

      \n
    1. Adapter is now set when defining Repo module and not in the configuration. (d3911953<\/code><\/a>)<\/li>\n
    2. Ecto.Multi.run\/3<\/code><\/a> now accepts a 2-arity function (first argument is now the Repo) instead of a 1-arity one before. (95d11cc2<\/code><\/a>)<\/li>\n<\/ol>\n

      Finally, there were a few minor glitches (or redundancies!) specific to Hex.pm: c4168977<\/code><\/a>, 21eb0bf8<\/code><\/a>, and 0929cd9e<\/code><\/a>.<\/p>\n

      Overall the update process was pretty straightforward. There were a few minor bugs along the way which were promptly fixed upstream. Having previously updated Hex.pm to Ecto 2.0<\/a>, which took a few months (we started it early on, which made it a fast moving target back then), I can really appreciate the level of maturity that Ecto achieved and how easy it was to update this time around. \ud83d\ude42<\/p>\n
